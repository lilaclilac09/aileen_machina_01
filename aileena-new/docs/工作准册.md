# Aileena Machina 工作准册

本文件是 **人 + Cursor + CI** 共用的工程标准。  
所有改动必须走完：**read → plan → small diff → verify → show evidence → iterate**

相关文件：
- Cursor 自动规则：`.cursor/rules/senior-engineer-loop.mdc`
- Prompt 快查：`.cursor/prompts.md`（指向本文 §5）
- DJ Set 自动化：`.github/workflows/verify-dj-set.yml`

---

## 1. 核心循环

```
read → plan → small diff → verify → show evidence → iterate
```

| 阶段 | 要求 |
|------|------|
| **read** | 先读相关文件；不猜结构、不脑补 |
| **plan** | 非 trivial 任务先给最小修改计划 |
| **small diff** | 只改必要代码；不重构无关部分 |
| **verify** | 跑可执行验证（命令 / 脚本 / 手测） |
| **show evidence** | 截图、录屏、`verify-report.json`、CI artifact |
| **iterate** | 失败则修，不口头宣称「应该可以」 |

**防乱改一句（最实用）：**

> 先不要写代码，先读相关文件并给我最小修改计划。

---

## 2. 编码守则

### 动手前

1. 不确定 → **列问题**，不要自行假设
2. 复杂逻辑 → 先写计划再实现
3. Bug → 先找根因再修

### 动手时

1. **最小必要 diff**
2. 保持现有风格、命名、抽象
3. **不引入新依赖**，除非说明必要性
4. **不改公共 API**，除非必须

### 收尾（每条任务必做）

1. 跑适用验证（见 §3）再标完成
2. 给出可复制验证步骤（命令、URL、点击路径）
3. UI / 交互 → **截图或录屏**，禁止只说「做完了」
4. Bug 修复 → 复述复现 → 根因 → 为何不影响其他路径
5. 列出 **风险** 和 **未解决问题**

### Evidence 原则（不可协商）

- 每条任务以 **verify + evidence** 结束，不是以意图总结结束
- 优先可运行检查，而非「应该 work」
- 跑不了验证（缺 cred / env）→ 明说原因 + 给用户可复制命令

## 3. 验证命令清单

在 `aileena-new/` 目录执行：

| 场景 | 命令 |
|------|------|
| 日常开发 | `pnpm dev` |
| 生产构建 | `pnpm build` |
| Lint | `pnpm lint` |
| **DJ Set / sound / carousel** | `pnpm verify:sound`（或 `verify:dj-set`，同一脚本） |
| 本地验证 | `pnpm verify:sound:local`（需 `pnpm dev -p 3011`） |
| 记忆同步 | `pnpm sync:content-memory` |
| 完整记忆工作流 | `pnpm memory:workflow` |

### 按改动类型选验证

| 改了什么 | 最少要跑 |
|----------|----------|
| UI / 页面 / 组件 | build + 相关页面截图或录屏 |
| `/sound` / `#dj-set` / deck carousel | `pnpm verify:sound` |
| 主页 Visual / `#glass-bench` | `pnpm verify:sound`（含主页检查） |
| API route | build + curl / 手测 |
| 脚本 / 记忆 | 对应 `pnpm` 脚本 + 输出文件检查 |
| 仅文档 | diff 自查 |

## 4. Sound 页面验收工作流

### 产品约定（不可违反）

| 项 | 标准 |
|----|------|
| **Deck carousel** | Handoff 5 首在 **DJ Station 内** `TrackLibraryBrowser`，`#dj-set` 锚点 |
| **Visual** | **只在主页** `#visual` / `#glass-bench`，**禁止**出现在 `/sound` |
| 曲目 | 仅 handoff 5 首（`lib/djSetlist.ts`） |
| 封面 | 真实发行 JPG，`public/dj-set/assets/covers/` |
| 旧路径 | `/dj-set` → 重定向 `/sound#dj-set` |
| 禁止 | 独立 `DJSetCarousel`；deck 30+ 首灌进 carousel；`/sound` 底部白块 Visual |

### Handoff 5 首

| TRACK | 曲目 | 封面来源 |
|-------|------|----------|
| DAYDRM | Daydreaming — Harry Styles | Spotify Harry's House |
| RAINFR | Rainforest — John Beltran · Open House | Bandcamp Now & Then |
| HIGHTD | High Tide — 同上 | 同上 |
| INTOUCH | In Touch — Beatrice M. | Apple / Tectonic Sinking |
| RNDVZ | Rendezvous — lovegold | Kynant Lovegod |

### 本地自动化验证

```bash
cd aileena-new
pnpm verify:sound              # 默认 production
pnpm verify:sound:local        # localhost:3011
```

**输出目录** `.verify-screenshots/`：

```
01-sound-top.png
02-sound-full-page.png          # 确认 /sound 无底部白块 Visual
03-deck-carousel-closeup.png
04-carousel-all-tracks.png
05-dj-set-redirect.png
06-home-visual.png
07-home-visual-closeup.png
sound-layout-walkthrough.webm
verify-report.json
```

**`verify-report.json` 检查项（含布局）：**

- `GET /sound`、`GET /` → 200
- `/dj-set` 重定向 → `/sound#dj-set`
- 5 张封面 JPG → 200
- **`/sound` 无 `#glass-bench`**
- **无 legacy `DJSetCarousel` section**
- Deck carousel 5 首 + 标题顺序 + 封面加载
- **`/#glass-bench` 主页 Visual 可见**

环境变量：

- `VERIFY_BASE_URL` — 默认 `https://www.aileena.xyz`
- `VERIFY_OUT_DIR` — 默认 `.verify-screenshots/`

### CI（GitHub Actions）

Workflow：`.github/workflows/verify-sound-layout.yml`

| 触发 | 说明 |
|------|------|
| push `main` | DJStation / GlassBench / sound / page.tsx 等 |
| schedule | 每天 07:30 UTC |
| workflow_dispatch | 可改 `base_url` |

Artifact 保留 30 天：截图 + webm + `verify-report.json`

### 录屏验收脚本（手测）

**`/sound`**

1. 打开 `/sound#dj-set` — 全黑，只有 DJ Station + deck 内 carousel
2. **底部不应有白色 Visual 区块**
3. Carousel 内 5 首 handoff，真实封面
4. `/dj-set` → 落到 `#dj-set`

**主页**

5. 打开 `/#glass-bench` — Visual（FROM THE KILN + polaroid）
6. Visual **不在** `/sound`

## 5. Cursor Prompt 模板

复制块 → 填 `[括号]` → 粘贴聊天。

### 0) 总控（最常用）

```
你是这个项目的 senior engineer。接下来所有修改遵守：
1 先读代码再判断
2 不确定先问
3 默认最小 diff
4 不做无关重构
5 不引入新依赖，除非说明必要性
6 每次修改后给验证步骤
7 对 ui/交互改动，说明如何截图或录屏验收
8 对复杂逻辑，先写计划再实现
9 对 bug，先找根因再修
10 输出风险和未解决问题

现在任务是：
[任务]
```

### 1) 需求澄清

```
先不要写代码。请阅读当前项目结构和相关文件，帮我把这个需求拆成：
1 目标
2 涉及模块
3 可能影响的文件
4 不明确的问题
5 最小实现方案
6 验证方式

需求是：
[贴需求]

如果有不确定的地方，先列问题，不要自行假设。
```

### 2) 定位问题

```
先不要修改代码。请根据下面的问题描述，在代码库中定位最可能相关的文件、函数、状态流和数据流。输出：
1 可能原因，按概率排序
2 需要查看的文件
3 需要加日志的位置
4 最小复现路径
5 你建议的修复方向

问题：[描述 bug]
复现步骤：[步骤]
期望：[期望]
实际：[实际]
```

### 3) 小 diff 实现

```
请实现这个需求，但遵守：
1 只做最小必要修改
2 不重构无关代码
3 不改公共 API，除非必须
4 保持现有风格
5 每个修改都说明原因
6 修改后告诉我如何验证

需求：[需求]
验收标准：[标准]
```

### 4) UI 截图确认

```
请根据这个截图/描述修改 ui。要求：
1 保持现有设计系统
2 不引入新组件库
3 只改相关组件
4 响应式不要破坏
5 输出修改点和验证路径

当前问题：[截图 + 描述]
目标效果：[截图/文字]
```

### 5) 交互 flow

```
请实现/修复这个交互流程。先分析状态机，再改代码。

流程：1 [步骤] 2 [步骤] 3 [步骤]
期望行为：[描述]
边界情况：[空状态/加载/错误/重复点击/取消/返回]

要求：
1 列出涉及状态
2 列出事件和状态转换
3 再做最小实现
4 给出手测步骤
```

### 6) 游戏 playable

```
请把这个玩法做成可测试版本，不追求最终美术，只追求核心 loop 可玩。

核心 loop：[玩家操作 → 系统反馈 → 奖励/失败 → 下一轮]

要求：
1 先实现最小 playable
2 明确输入方式
3 明确胜负/完成条件
4 加基础 debug 信息
5 给我本地运行方式
6 列出下一版可调参数

不要做大而全系统，先让它能玩。
```

### 7) 视频验收

```
请帮我生成一个适合录屏验收的测试脚本。包括：
1 场景名称
2 初始状态
3 操作步骤
4 每一步预期画面
5 需要重点观察的状态变化
6 失败时如何判断

功能：[功能描述]
```

自动化：`pnpm verify:sound` → `.verify-screenshots/`

### 8) Bug 修复

```
请修复这个 bug。要求：
1 先复述复现路径
2 找到根因
3 只做最小修复
4 如果有测试框架，补一个回归测试
5 如果没有测试框架，给出手动验证步骤
6 说明为什么这个修复不会影响其他路径

bug：[描述]
复现：[步骤]
期望：[期望]
实际：[实际]
```

### 9) Code review

```
请 review 当前 diff。重点检查：
1 是否满足需求
2 是否有过度实现
3 是否破坏现有行为
4 状态/异步/错误处理是否安全
5 类型是否合理
6 命名是否清楚
7 有没有可删代码
8 有没有隐藏 bug

请按 blocking / non-blocking / nit 输出。
```

### 10) 合并前检查

```
请做合并前检查。不要改代码，先输出风险清单：
1 需求是否完成
2 测试是否覆盖
3 是否有 console/debug code
4 是否有 hardcoded 临时数据
5 是否有性能风险
6 是否有安全/权限问题
7 是否有迁移/兼容问题
8 需要我人工确认的地方
```

### 11) 文档沉淀

```
请根据这次修改生成一段工程记录，包含：
1 背景
2 修改了什么
3 为什么这样做
4 影响范围
5 验证方式
6 已知 tradeoff
7 后续 todo

要求简洁，可直接放进 pr description / changelog。
```
